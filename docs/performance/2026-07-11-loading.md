## Representative baseline

One warm-up followed by five Release measurements used seed `262`, Minecraft `1.18`, the Overworld, a `128 x 128` biome grid at scale `1`, and structure rectangle `[-242, -182, 242, 182]`. The rectangle models a 800 x 600 map view at 2 pixels per block; all 23 supported Overworld structure types were selected.

| Stage | Median | Maximum | Result count |
| --- | ---: | ---: | ---: |
| Cubiomes biome grid | 5.846 ms | 11.551 ms | 16,384 biome IDs |
| Structures, one Core call with all types | 731.534 ms | 859.520 ms | 119 |
| Structures, AppKit-equivalent one Core call per type | 733.793 ms | 748.798 ms | 119 |

Combining all selected types into a single `CubiomesCore.structures` call improved the representative structure query by only about 0.3%. It was rejected as a meaningful optimization.

## Qt/AppKit path mapping

| Concern | Qt reference | SwiftBiomes before this change | Outcome |
| --- | --- | --- | --- |
| Chunk loading | `src/world.cpp` `Level::resizeLevel` reuses cached quads | `BiomeMapRenderer` uses batched grids, while `BiomeTileCache` used string keys and a linear LRU update for every hit | Use structured keys and constant-time hits; retain 512-entry bound and LRU eviction. |
| Structure loading | `src/world.cpp` `Level::update` keeps padded structure cells until the view exits them | `BiomeMapView.requestVisibleStructures` queried a new exact padded rectangle after any pan and cancelled the previous task | Reuse one bounded 20%-padded coverage result until the visible rect leaves it. |
| Scheduling | `MapWorker::run` processes prioritized reusable quads | Tile work remains user-initiated and centre-first; structure work remains serial and cancellable | No semantic or public-interface change. |

## Confirmed bottlenecks and changes

1. `BiomeMapView.drawOverlayNotice` calls `requestVisibleStructures` on every redraw. Before this change, the exact visible rectangle was part of the only request key, so a pan changed the key, cancelled work, and re-ran the 733.793 ms representative query. `StructureOverlayCacheEntry` now holds one settings/type-specific padded coverage result and avoids a new query while the unpadded visible rect remains inside it. The cache is cleared on settings/type refresh and cannot grow without bound.
2. `BiomeTileCache.image(for:)` previously allocated a string key and linearly edited an access-order array for every visible tile cache hit. It now indexes by `BiomeMapTileKey` and updates a monotonic access counter, doing eviction scans only on insertion past the fixed limit. Pending-structure de-duplication also uses its existing value-type key rather than a newly built string; pending-tile tracking remains string-based because the completion crosses an isolation boundary.

Both changes preserve generation, filtering, cancellation, selected types, and rendered content. The new tests cover coverage identity/bounds and LRU recency.

## Stronghold bottleneck and reuse

Runtime sampling of the real Release overlay request showed the worker thread in `CubiomesWorld.structures(types:rect:)`, specifically its stable sort. A per-type Release probe using the actual UI coverage `[-272, -176, 272, 176]` established the dominant cause: Stronghold had a 730.296 ms median (731.067 ms maximum), while every other supported Overworld type was below 0.1 ms. The combined 23-type Core call measured 733.137 ms median. Trace: `/private/tmp/SwiftBiomes-structure-loading.sample.txt`.

`CubiomesWorld.strongholds()` independently enumerates the same bounded global list that the dependency's rectangle query enumerates before filtering. `StrongholdLocationsCache` retains one settings-keyed list, then filters it by the requested rectangle in the same `(Z, X)` order. It replaces the cached entry when seed, version, or dimension changes and is bounded to at most 128 locations. The regression test compares overlay stronghold coordinates against the original `CubiomesCore.structures` result.

One real Release UI cold query at the same view completed in 1.066 s from the `None → All` type action to the Inspector completion state. A following warm-query automation attempt did not return a usable measurement before the external automation environment became unavailable, so no warm-latency improvement is claimed yet. Earlier UI timings collected by a script that unintentionally changed zoom are rejected; that script was corrected before the cold measurement.

## Instrumentation

Release profiling can use Instruments' Points of Interest template. The app emits `OSSignposter` intervals with subsystem `com.LloydME.SwiftBiomes`: category `MapLoading`, interval `Render Tile`; and category `StructureLoading`, interval `Load Structures`. These boundaries cover the synchronous renderer and Cubiomes structure-provider work respectively, so a future visible-window run can collect median/P95 latency, CPU time, allocations, and memory under the fixed scenario without adding profiling-only code.

## Runtime launch prerequisite

The original Release application could launch without a visible window. The sole initial `NSWindowController` in `Base.lproj/Main.storyboard` had `visibleAtLaunch="NO"`, while `AppDelegate.applicationDidFinishLaunching` does not show a window. This was corrected to `visibleAtLaunch="YES"`; macOS accessibility later observed one unhidden, unminimized 900 × 580 window and the real structure completion state. The multi-run automation that followed remains unreliable, as noted below.

## Remaining measurement gap

No aligned Qt map/structure latency baseline is available yet. The installed Qt executable's `--nogui` mode measures seed search only, not map tiles or structure overlays. The Swift Release UI window and one cold structure workflow were observed successfully, but the external automation environment later stopped returning a warm-query result and then denied further process/log inspection. Cold/warm distributions, pan, cancellation, task switching, chunk time-to-first-tile, main-thread stalls, allocations, and matching Qt measurements remain required before claiming parity.
