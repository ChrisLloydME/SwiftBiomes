## Root causes and retained changes

### Structure loading

The cold all-types Swift request originally published nothing until the serial provider finished Stronghold enumeration. A per-type probe showed a 730.296 ms Stronghold median while every other Overworld type was below 0.1 ms. Qt schedules Stronghold separately and publishes recurring-structure tiles as workers finish.

Swift now makes a fast non-Stronghold request first when Stronghold is part of a multi-type selection, publishes those points while status remains `loading`, then executes the original full-type request and caches its unchanged final result. Generation and key guards prevent stale UI commits, and cancellation is checked again before entering the expensive full request. A regression test covers the fast/full type plan.

This changed first-result latency from 718.204 ms to 0.929 ms. The final full result, filtering, ordering, status, selection, and public interfaces are unchanged.

`StrongholdLocationsCache` also retains one settings-keyed list of at most 128 locations from `CubiomesWorld.strongholds()` and filters that list by the requested rectangle. A regression test compares the coordinates against the original `CubiomesCore.structures` result.

### Structure coverage reuse

`drawOverlayNotice` can call `requestVisibleStructures` on every redraw. The previous exact-rectangle key caused a pan to cancel and repeat the full query. `StructureOverlayCacheEntry` now retains one settings/type-specific 20%-padded coverage result until the visible rect exits that coverage. The cache is cleared on settings or type refresh and cannot grow without bound.

### Tile-cache lookup and visible working set

`BiomeTileCache` previously allocated a string key and linearly edited an access-order array on each hit. It now indexes by `BiomeMapTileKey` and uses a monotonic access counter, so hits are constant-time and eviction scanning occurs only on insertion over capacity.

A real full-screen, minimum-scale run exposed a correctness bug in the fixed 512-tile capacity: the padded active layer exceeded 512 tiles. Centre-first draw traversal made centre tiles the least recently touched entries; insertion then evicted and immediately re-requested them forever. The result was random centre white holes and sustained 526–640% CPU.

Before iterating a layer, the view now ensures the cache can hold twice that layer's current visible working set. This reserves no image memory by itself; it only prevents eviction while the current visible layers are being assembled. The full-screen reproduction now settles to 0% CPU instead of looping, and the new cache test verifies an expanded visible working set remains resident.

## Rejected chunk optimization and pause decision

A single-variable experiment stopped requesting missing distant fallback levels and kept only active plus adjacent fallback work. It reduced the aligned cold workload from 40 to 28 tiles and the median from 215.854 ms to 79.712 ms with four workers. Matching Qt's eight workers produced a 69.100 ms median and 83.085 ms P95, meeting the Qt timing target.

That change was rejected and reverted. In the real app, rapid scale changes could cancel the currently covering levels before a replacement covered the dirty region. Because `draw(_:)` clears the dirty rect first, the user saw random white gaps. The offending commit `53a7389` was reverted by `c498826`; the safe cache-working-set fix is `904b177`.

This is the strict pause condition for chunk parity: two evidence-based cache/scheduling iterations identified the extra fallback work, but the only measured parity configuration worsened visible behaviour. Further progress requires a larger design change such as a retained composited backing surface, explicit visible-tile pinning plus priority queues, or cancellable per-scale prefetch. It should not be attempted as another local scheduling tweak.

## Memory

The first Swift command-line runner accumulated autoreleased AppKit bitmap objects and reported 85.8 MB RSS. With an autorelease pool per simulated event cycle, peak RSS is 35.5 MB versus Qt's 31.6 MB. The 12.3% RSS difference is explained by fixed framework residency: minimal Swift/AppKit and Qt/Widgets programs measured 27.5 MB and 19.4 MB respectively. Workload-incremental RSS is therefore about 8.0 MB for Swift versus 12.2 MB for Qt.

macOS peak memory footprint for the aligned workload was 13.0 MB for Swift and 14.4 MB for Qt. There is no unexplained workload memory regression over 10%.

The full-screen cache fix may retain more tiles when the visible layer itself exceeds 512 entries. This is intentional correctness memory, bounded by the largest observed visible layer during the session rather than unbounded per-pan growth.
